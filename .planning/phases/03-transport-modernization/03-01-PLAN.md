  <name>Task 1: Add transport-default helpers, additive profile metadata, and backward-compatible metadata migration</name>
  <files>xrayebator</files>
  <action>
1. Add compact helper functions near the other helper utilities, before the menu section. Keep them Bash-simple and local to this phase:
   - `find_available_random_port()` or equivalent: generate a random port in the inclusive range `30000-60000` and retry until no existing inbound in `config.json` uses that port.
   - `generate_transport_token()` or equivalent: use `openssl rand -hex` to create ASCII-safe randomized suffixes for transport-specific values.
   - `build_transport_defaults()` or equivalent: for a given transport, return the default port plus optional `grpc_service_name` / `xhttp_path`.
   - `read_profile_transport_metadata()` and `write_profile_transport_metadata()` or equivalent: one helper seam for reading and persisting `grpc_service_name` / `xhttp_path` so `create_profile()`, `add_inbound()`, `generate_connection()`, and `change_port_menu()` do not each hand-roll transport metadata handling.
   - `confirm_vision_443_risk()` or equivalent: show an explicit Russian warning that Vision on `443` is often blocked by TSPU in 2026 and require positive confirmation (`y/Y/д/Д`) before continuing.

2. Expand the profile metadata schema in `create_profile()` to store transport extras additively:
   - Add optional fields `grpc_service_name` and `xhttp_path` to newly created profile JSON when relevant.
   - Do not remove or rename existing fields. Old profiles must continue to load unchanged.
   - Keep the project’s current file-creation style for new profiles, but use `safe_jq_write` for profile updates to existing files.

3. Expand `add_inbound()` so transport-specific values are passed in explicitly rather than hardcoded internally:
   - Update the function signature to accept `grpc_service_name` and `xhttp_path`.
   - In the gRPC inbound template, replace hardcoded `"serviceName": "grpc"` with the passed profile/inbound value.
   - In the XHTTP inbound template, replace hardcoded `"path": "/xhttp"` with the passed profile/inbound value.
   - Keep XHTTP `host` tied to the current SNI as today.

4. Preserve shared inbound semantics when adding a profile to an already existing inbound on the same port:
   - If transport is gRPC and the inbound already exists, read the actual inbound `grpcSettings.serviceName` and update the just-created profile JSON to that value.
   - If transport is XHTTP and the inbound already exists, read the actual inbound `xhttpSettings.path` and update the just-created profile JSON to that value.
   - This must happen alongside the existing SNI/fingerprint synchronization logic so two profiles on one inbound never export mismatched transport metadata.

5. Add one metadata-only migration/backfill in `main_menu()` with a new marker file, following the project’s existing migration pattern:
   - Suggested function name: `migrate_transport_profile_metadata()`.
   - Hook location: the startup migration block in `main_menu()`, alongside the existing XHTTP/config migrations and before the final `needs_restart` restart section.
   - Marker file: `/usr/local/etc/xray/.transport_profile_metadata_migrated`.
   - Safety mechanics: call `backup_config "transport_profile_metadata"` before any profile rewrite, use `safe_jq_write` for each profile update, call `fix_xray_permissions` after writes, and only use `safe_restart_xray` if the implementation ends up mutating `config.json` as part of the migration.
   - Scope: iterate profile JSON files; if `transport == "grpc"` and `grpc_service_name` is missing, write `"grpc"`; if `transport == "xhttp"` and `xhttp_path` is missing, write `"/xhttp"`.
   - This migration should preserve compatibility only. Do not randomize existing live profiles in place.
   - Because this migration changes only profile metadata, do not require Xray restart unless the implementation also mutates `config.json`.
  </action>

  <name>Task 2: Rework create/change/connect flows around modern transport defaults and honest TSPU guidance</name>
  <files>xrayebator</files>
  <action>
1. Rewrite `create_profile_menu()` transport order and copy so the user-facing default is modern and honest:
   - Make `VLESS + XHTTP + Reality` the first option and mark it as recommended.
   - Update the Russian descriptions to reflect current 2026 TSPU guidance, especially which options are generally safer on mobile networks versus wired/home ISP links.
   - Downgrade Vision-on-443 style choices from implicit defaults to clearly risky/manual options.
   - Keep the menu compact; do not turn it into a large architecture refactor.

2. Rework `create_profile_menu()` -> `create_profile()` call sites so new profiles no longer pass fixed low ports:
   - Replace hardcoded defaults `443`, `8443`, `2053`, `9443` as the standard creation path with a helper-generated random available high port.
   - gRPC profiles must also receive a randomized `grpc_service_name`.
   - XHTTP profiles must also receive a randomized `xhttp_path`.
   - Vision-based transports can still be offered, but if the selected port is `443`, the flow must call `confirm_vision_443_risk()` before any profile or inbound write.
   - Keep the current gRPC-specific SNI handling and existing SNI list behavior unless transport-specific requirements demand otherwise.

3. Update `generate_connection()` / connection output so exported links and manual instructions use persisted metadata instead of hardcoded literals:
   - For gRPC, read `grpc_service_name` from the profile and fall back to `"grpc"` only when missing.
   - For XHTTP, read `xhttp_path` from the profile and fall back to `"/xhttp"` only when missing.
   - Ensure manual output (`serviceName`, `Path`) and VLESS links stay consistent with the inbound config created earlier.

4. Update `change_port_menu()` so it does not steer users back to deprecated defaults:
   - Make random high port the first/default recommendation.
   - Leave low HTTPS-like ports available only as explicit/manual choices.
   - If a Vision transport is being changed to `443`, require the same `confirm_vision_443_risk()` acknowledgement used during profile creation.
   - Preserve the existing inbound/firewall/update logic after the port choice is accepted.
   - If a profile is moved onto an already existing inbound, the profile must adopt that inbound’s transport metadata instead of generating fresh values.
   - If an inbound is moved to a new port, every profile that remains on that inbound must keep the same transport metadata it had before the move.

5. Keep backward compatibility for older profiles and older menu paths:
   - Existing non-random gRPC/XHTTP profiles should still export working links through the fallback values.
   - No new hardcoded `grpc` or `/xhttp` assumptions should remain in the fresh profile creation path.
   - All new or changed Russian user-facing strings must match the project’s existing style.
  </action>
